# AI Flashcards — Design Spec

**Date**: 2026-06-09  
**Project**: MIR Prep  
**Status**: Approved for implementation

---

## 1. Overview

A new section in the app where authenticated users can upload a PDF or Word document, choose a medical exam type and flashcard count, and have Claude generate flashcards from the document content. The user reviews each generated card, keeps or discards individually, and saves only the approved ones to their dashboard. Saved flashcards can be downloaded as a print-ready double-sided PDF at any time.

---

## 2. Navigation

A new entry **"FLASHCARDS IA"** is added to the main NavBar at the same level as Dashboard, Especialidades, and Simulacro. Route: `/flashcards`.

No sub-pages. API key configuration is a modal inside `/flashcards`.

---

## 3. User Flow

```
1. User opens /flashcards
2. If no Claude API key configured → prompt to add it (modal)
3. User uploads PDF or DOCX file
4. User selects:
   - Count: 10 / 20 / 30 flashcards
   - Exam type: MIR | ENARM | U. del Rosario | U. del Bosque | USMLE | Personalizado
5. Click "GENERAR FLASHCARDS"
6. Loading state while Claude processes
7. Preview screen: all cards shown in 9:16 flip format, NOT saved yet
   - User can flip each card to review front/back
   - Below the grid: two small buttons — "Guardar estas flashcards" / "Eliminar estas flashcards"
8. Click "Guardar estas flashcards" → entire batch saved to DB at once
9. Redirect to dashboard section within /flashcards showing all saved cards
10. "DESCARGAR PDF" button → on-demand PDF generation, filtered by specialty or exam type
```

---

## 4. Flashcard Data Model

### Addition to `Usuario` model
```prisma
claudeApiKeyEnc  String?   // AES-256-GCM encrypted, format: "iv:tag:ciphertext"
```

### New `Flashcard` model
```prisma
model Flashcard {
  id           String   @id @default(cuid())
  pregunta     String
  respuesta    String
  especialidad String   // auto-detected by Claude from document
  tipoExamen   String   // MIR, ENARM, U.Rosario, U.Bosque, USMLE, or custom
  consejo      String   // study tip generated by Claude, phrased for the selected exam type
  usuarioId    String
  usuario      Usuario  @relation(fields: [usuarioId], references: [id], onDelete: Cascade)
  creadoEn     DateTime @default(now())
}
```

## 5. API Routes

| Route | Method | Auth | Purpose |
|-------|--------|------|---------|
| `/api/user/claude-key` | POST | Required | Save or update encrypted API key |
| `/api/user/claude-key` | GET | Required | Returns `{ configured: boolean }` only — never returns the key |
| `/api/flashcards/generate` | POST multipart | Required | Extract text, call Claude, return preview array (does NOT save to DB) |
| `/api/flashcards/save` | POST JSON | Required | Save approved flashcards array to DB |
| `/api/flashcards` | GET | Required | Return user's saved flashcards (filters: especialidad, tipoExamen) |
| `/api/flashcards/[id]` | DELETE | Required | Delete a single saved flashcard |
| `/api/flashcards/pdf` | GET | Required | Generate and return PDF (query params: especialidad?, tipoExamen?) |

## 6. Security — API Key Encryption

**Algorithm**: AES-256-GCM (authenticated encryption, tamper-proof)  
**Encryption key**: `FLASHCARD_ENCRYPTION_KEY` environment variable (32-byte hex string), set in Vercel and local `.env`.

**Storage format in DB**: `"<iv_hex>:<authTag_hex>:<ciphertext_hex>"`

**Rules**:
- The key is decrypted in memory only at call time inside `/api/flashcards/generate`, used once, then discarded
- The decrypted key is never logged, never returned to the client, never stored in any intermediate variable beyond the Anthropic SDK call
- The GET `/api/user/claude-key` endpoint returns only `{ configured: true/false }`
- If decryption fails (corrupted data, wrong env key), the endpoint returns 400 with a generic error and prompts the user to re-enter their key

**Utility**: `src/lib/crypto.ts` — two exported functions: `encryptApiKey(plain: string): string` and `decryptApiKey(stored: string): string`.

## 7. Claude Integration

**Model**: `claude-sonnet-4-6` (cost-effective for generation; user can upgrade to Opus in settings later)

**System prompt** (parameterized at request time):

```
Eres un experto en preparación para el examen {tipoExamen}.
Genera exactamente {count} flashcards médicas en formato JSON a partir del texto del documento.

Cada flashcard debe ser un objeto con exactamente estos campos:
- "pregunta": pregunta concisa, estilo {tipoExamen}, que evalúe un concepto clave
- "respuesta": respuesta directa, completa y clara
- "especialidad": especialidad médica detectada del contexto (ej. "Cardiología", "Neurología")
- "consejo": consejo de estudio específico para {tipoExamen}, empezando con "En el {tipoExamen}..."

Responde ÚNICAMENTE con un array JSON válido. Sin markdown, sin explicaciones, sin texto adicional.
```

**Document extraction**:
- PDF: `pdf-parse` (Node.js, server-side)
- DOCX: `mammoth` (Node.js, server-side)
- Max file size: 10MB
- Text truncated to 80,000 characters before sending to Claude (covers most study documents)

**Response handling**: Parse JSON response, validate that each object has all 4 required fields. If parsing fails, return 422 with a user-friendly error. The `tipoExamen` field on each saved Flashcard is copied from the user's selection, not from Claude's output.

## 8. Flashcard Visual Design

**Format**: 9:16 aspect ratio (54mm × 96mm), portrait orientation.

**Front** (cream background `#f5f0e8`, ink text `#1a1a18`):
- Top: specialty tag (monospace, small caps) + colored accent bar (`#E84A1F`)
- Center: question text (serif, bold, centered)
- Bottom: "PREGUNTA" label + ✦ watermark

**Back** (ink background `#1a1a18`, cream text `#f5f0e8`):
- Top: specialty tag + accent bar (same as front)
- Center: answer text + TIP MIR section (separated by divider, smaller text, muted)
- Key terms highlighted in orange `#E84A1F`
- Bottom: "RESPUESTA" label + ✦ watermark

**Interaction**: Click card to flip (CSS 3D transform, 0.4s ease). On preview screen, each card also shows Guardar/Descartar buttons below it. On dashboard, cards are flippable only.

---

## 9. PDF Export

Generated server-side with `pdfkit`. On-demand only — triggered by user clicking "DESCARGAR PDF".

**Layout on A4 page**:
- 3 columns × 4 rows = 12 cards per page
- Card size: 54mm × 96mm
- Margins: 10mm all sides, 4mm gutter between cards
- Dashed cut guides on all card borders

**Double-sided alignment**:
- Page 1 (fronts): columns in order [1, 2, 3], [4, 5, 6]...
- Page 2 (backs): columns **mirrored** [3, 2, 1], [6, 5, 4]... so that flipping the paper horizontally aligns each back with its front
- Page header: username + specialty/exam filter + generation date

**Filtering**: User can download all saved cards, or filter by `especialidad`, or filter by `tipoExamen`. Each combination generates its own PDF on demand.

## 10. Page Layout — `/flashcards`

Two visual states on the same route:

**State A — Generator** (default, or when no saved cards):
- Upload dropzone (PDF/DOCX, drag & drop or click)
- Count selector: `10 / 20 / 30` pill buttons
- Exam type dropdown + optional custom text input
- "GENERAR FLASHCARDS →" primary button
- API key status indicator + "Configurar" link

**State B — Preview** (after generation, before saving):
- Grid of all generated cards in 9:16 flip format (user can flip each to review)
- Single small action bar below the grid with two buttons:
  - "Guardar estas flashcards" → saves the entire batch to DB
  - "Eliminar estas flashcards" → discards the batch, returns to State A
- No per-card actions — decision is made for the whole batch at once

**State C — Dashboard** (after saving, persistent):
- Filter bar: by especialidad, by tipoExamen
- Grid of saved flip cards
- "DESCARGAR PDF" button (respects active filters)
- "GENERAR MÁS" button to return to State A
- Individual delete (trash icon) per card

## 13. Out of Scope

- Re-generating from a previously uploaded file (file is not stored)
- Editing flashcard content after generation
- Sharing flashcard sets between users
- Flashcard study mode with spaced repetition
- Upgrading/downgrading Claude model per-user
